# MCP Tools

Engram exposes **15 MCP tools** for memory management. All tools work via stdio transport and are compatible with any MCP client.

---

## Core Memory Tools

### `mem_save`

Save a structured observation (decision, bugfix, pattern, etc.)

**Parameters:**
- `title` (string, required) — Observation title
- `type` (string, required) — Type: bugfix, architecture, decision, pattern, config, discovery, learning
- `content` (string, required) — What/Why/Where/Learned format
- `scope` (string, optional) — `project` (default) or `personal`
- `topic_key` (string, optional) — Stable key for evolving topics
- `project` (string, optional) — Override detected project name

**Example:**
```json
{
  "title": "Fixed N+1 query in user list",
  "type": "bugfix",
  "content": "What: User list was making N+1 database queries\nWhy: Performance issue causing slow page loads\nWhere: src/internal/store/user.go:142\nLearned: Use Include() for eager loading in GORM"
}
```

---

### `mem_update`

Update an existing observation by ID

**Parameters:**
- `id` (number, required) — Observation ID to update
- `title` (string, optional) — New title
- `type` (string, optional) — New type
- `content` (string, optional) — New content
- `scope` (string, optional) — New scope

**Example:**
```json
{
  "id": 42,
  "content": "What: User list was making N+1 queries\nWhy: Performance issue\nWhere: src/internal/store/user.go:142\nLearned: Use Include() for eager loading in GORM\nNote: Also added caching layer"
}
```

---

### `mem_delete`

Delete an observation (soft-delete by default, hard-delete optional)

**Parameters:**
- `id` (number, required) — Observation ID to delete
- `hard` (boolean, optional) — If true, permanently delete; if false, soft-delete (default)

**Example:**
```json
{
  "id": 42,
  "hard": false  // soft-delete (default)
}
```

**Soft-delete behavior:**
- Sets `deleted_at` timestamp
- Excluded from search results
- Can be recovered with `mem_get_observation`

## Search & Navigation Tools

### `mem_search`

Full-text search across all memories

**Parameters:**
- `query` (string, required) — Search query (FTS5 matches)
- `project` (string, optional) — Filter by project
- `scope` (string, optional) — Filter by scope (`project` or `personal`)
- `type` (string, optional) — Filter by type
- `limit` (number, optional) — Max results (default: 20)
- `include_deleted` (boolean, optional) — Include soft-deleted (default: false)

**Example:**
```json
{
  "query": "auth middleware",
  "project": "engram-landing",
  "limit": 10
}
```

**Progressive Disclosure Pattern:**
1. Search returns compact results (~100 tokens each)
2. Use `mem_timeline` for session context
3. Use `mem_get_observation` for full content

---

### `mem_timeline`

Get chronological context around a specific observation

**Parameters:**
- `observation_id` (number, required) — Target observation ID
- `before` (number, optional) — Include N observations before (default: 3)
- `after` (number, optional) — Include N observations after (default: 3)

**Example:**
```json
{
  "observation_id": 42,
  "before": 5,
  "after": 5
}
```

Returns:
- Target observation
- Observations from same session before
- Observations from same session after

---

### `mem_get_observation`

Get full content of a specific memory

**Parameters:**
- `id` (number, required) — Observation ID

**Example:**
```json
{
  "id": 42
}
```

Returns full untruncated content, metadata, and timeline.

## Session Management Tools

### `mem_session_summary`

Save end-of-session summary

**Parameters:**
- `goal` (string, required) — Session objective
- `discoveries` (array, required) — Key findings
- `accomplished` (array, required) — What was completed
- `files` (array, required) — Files touched/modified
- `next_steps` (string, optional) — What to do next session

**Example:**
```json
{
  "goal": "Fix N+1 query issue in user list",
  "discoveries": [
    "User list was loading each user separately",
    "Database was making 150 queries for 150 users"
  ],
  "accomplished": [
    "Added Include() for eager loading",
    "Reduced queries from 150 to 1",
    "Page load time reduced from 2.5s to 150ms"
  ],
  "files": [
    "src/internal/store/user.go",
    "src/internal/api/user.go"
  ],
  "next_steps": "Add caching layer for frequently accessed users"
}
```

---

### `mem_context`

Get recent context from previous sessions

**Parameters:**
- `count` (number, optional) — Number of recent sessions (default: 5)
- `project` (string, optional) — Filter by project

**Example:**
```json
{
  "count": 10,
  "project": "engram-landing"
}
```

Returns:
- Recent session summaries
- Key decisions made
- Files modified
- Open questions

**Use after compaction** to recover session state.

---

### `mem_session_start`

Register a session start

**Parameters:**
- `goal` (string, optional) — Session objective
- `project` (string, optional) — Project name

**Example:**
```json
{
  "goal": "Add dark mode to landing page",
  "project": "engram-landing"
}
```

Automatically adds timestamp and creates session context.

---

### `mem_session_end`

Mark a session as completed

**Parameters:**
- `duration` (number, optional) — Session duration in seconds

**Example:**
```json
{
  "duration": 3600  // 1 hour
}
```

Used to track session statistics.

## Metadata & Statistics Tools

### `mem_stats`

Get memory system statistics

**Parameters:**
None

**Returns:**
- Total observations
- Total sessions
- Total prompts
- Observations per project
- Observations per type
- Recent activity timeline

---

### `mem_suggest_topic_key`

Suggest a stable topic_key for evolving topics

**Parameters:**
- `type` (string, required) — Type of topic
- `title` (string, required) — Topic title
- `history` (string, optional) — Previous observations on this topic

**Example:**
```json
{
  "type": "architecture",
  "title": "Auth architecture",
  "history": "Previous auth decisions and patterns"
}
```

**Returns:**
- Suggested topic_key (e.g., `architecture-auth-architecture`)
- Family heuristic applied based on type

**Use before `mem_save`** to ensure topic_key consistency across sessions.

## Advanced Tools

### `mem_save_prompt`

Save a user prompt for future context

**Parameters:**
- `prompt` (string, required) — User's prompt
- `type` (string, optional) — Prompt type (question, instruction, etc.)
- `context` (string, optional) — Additional context

**Example:**
```json
{
  "prompt": "How do I fix the N+1 query issue?",
  "type": "question",
  "context": "User asked about database performance optimization"
}
```

---

### `mem_capture_passive`

Extract learnings from text output

**Parameters:**
- `text` (string, required) — Text to analyze
- `type` (string, optional) — Inferred type
- `project` (string, optional) — Project name

**Example:**
```json
{
  "text": "User list optimization: Changed from N+1 queries to Include() call. Reduced page load from 2.5s to 150ms.",
  "type": "learning"
}
```

Automatically extracts title, type, and What/Why/Where/Learned format.

---

### `mem_merge_projects`

Merge project name variants into canonical name (admin)

**Parameters:**
- `source_project` (string, required) — Project name to merge
- `target_project` (string, required) — Canonical project name

**Example:**
```json
{
  "source_project": "engram-landing",
  "target_project": "engram-landing-page"
}
```

**Use case:** When project names are inconsistent across sessions.

## Progressive Disclosure Pattern

For token-efficient memory retrieval, use this 3-layer pattern:

### Layer 1: `mem_search`
```
Search for "auth middleware" → returns ~10 compact results
```

### Layer 2: `mem_timeline`
```
Get context around observation_id=42 → shows related sessions
```

### Layer 3: `mem_get_observation`
```
Get full content of id=42 → untruncated details
```

This pattern:
- Avoids dumping everything at once
- Reduces token usage
- Makes context more relevant

## Error Handling

All tools return structured errors:

```json
{
  "error": "Observation not found",
  "id": 42,
  "type": "not_found"
}
```

**Common errors:**
- `not_found` — ID doesn't exist
- `invalid_format` — Content doesn't match What/Why/Where/Learned
- `duplicate` — Exact duplicate detected
- `conflict` — Observation modified since last read
- `forbidden` — Cannot delete/update (permissions)

## Best Practices

1. **Save proactively** — Don't wait to be asked
2. **Use topic_key** — For evolving topics
3. **End sessions** — Always call `mem_session_end`
4. **Recover context** — Call `mem_context` after compaction
5. **Search first** — Use progressive disclosure pattern
6. **Update revisions** — Use `mem_update` for evolving decisions
7. **Soft-delete** — Let `mem_delete` soften-delete by default
